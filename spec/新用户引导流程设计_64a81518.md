# 新用户引导流程设计方案

## 设计理念

基于onboarding-cro skill的核心原则:
- **Time-to-Value优先**: 最短路径达到核心价值(成功切换供应商)
- **Progress Creates Motivation**: 可视化进度激励用户完成
- **Empty State = Opportunity**: 空状态是引导的最佳时机
- **Checklist Pattern**: 3-5步,有序但可跳跃,显示完成进度

## 推荐方案: 方案2(紧凑式)

**理由:**
1. **不干扰主流程**: 可以轻松关闭,不强制用户跟随
2. **进度清晰**: 进度条+步骤计数(1/3),一目了然
3. **行动导向**: 每个步骤都有明确的CTA按钮
4. **响应式友好**: 占用空间小,在窄屏下也能良好显示

// 使用Tauri的Command API执行shell命令
import { Command } from '@tauri-apps/api/shell';
import { open } from '@tauri-apps/api/shell';

// 智能安装流程
export async function install(options: InstallOptions) {
  const { appId, onProgress, onComplete, onError } = options;
  const command = getInstallCommand(appId);
  
  try {
    // 步骤1: 检测权限
    onProgress?.(5, '检测安装权限...');
    const hasPermission = await checkPermission();
    
    if (!hasPermission) {
      // 权限不足，降级到手动安装
      onError?.({
        code: 'PERMISSION_DENIED',
        message: '需要管理员权限，请使用手动安装方式',
        fallbackAction: 'manual'
      });
      return;
    }
    
    onProgress?.(10, '开始安装...');
    
    // 执行安装命令
    const child = new Command('sh', ['-c', command]);
    
    child.on('close', (data) => {
      if (data.code === 0) {
        onProgress?.(100, '安装完成！');
        onComplete?.();
      } else if (data.code === 126 || data.code === 127) {
        // 权限错误，提示降级
        onError?.({
          code: 'PERMISSION_DENIED',
          message: '安装需要管理员权限',
          fallbackAction: 'manual'
        });
      } else {
        onError?.({
          code: 'INSTALL_FAILED',
          message: `安装失败，退出码: ${data.code}`,
          fallbackAction: 'retry'
        });
      }
    });
    
    child.stdout.on('data', (line) => {
      onProgress?.(50, line); // 解析实际进度
    });
    
    child.stderr.on('data', (line) => {
      console.error('安装错误:', line);
      // 检测常见权限错误
      if (line.includes('EACCES') || line.includes('permission denied')) {
        onError?.({
          code: 'PERMISSION_DENIED',
          message: '权限不足，请使用手动安装',
          fallbackAction: 'manual'
        });
      }
    });
    
    await child.spawn();
    
  } catch (error) {
    onError?.({
      code: 'UNKNOWN_ERROR',
      message: error.message,
      fallbackAction: 'manual'
    });
  }
}

// 检测是否有npm全局安装权限
export async function checkPermission(): Promise<boolean> {
  try {
    const testCommand = new Command('sh', ['-c', 'npm root -g']);
    const output = await testCommand.execute();
    return output.code === 0;
  } catch {
    return false;
  }
}

// 打开系统终端并自动粘贴命令
export function openTerminalInstall(appId: AppId): void {
  const command = getInstallCommand(appId);
  
  // macOS: 打开Terminal.app
  if (navigator.platform.includes('Mac')) {
    open(`osascript -e 'tell application "Terminal" to do script "${command}"'`);
  }
  // Windows: 打开PowerShell
  else if (navigator.platform.includes('Win')) {
    open(`powershell -Command "Set-Clipboard -Value '${command}'; Start-Process powershell"`);
  }
  // Linux: 打开默认终端
  else {
    open(`x-terminal-emulator -e "${command}"`);
  }

### 自动完成检测

步骤完成状态分两种策略：

| 步骤 | 完成判断方式 | 持久化 | 删除后行为 |
|------|------------|--------|----------|
| Step 1: 安装 CLI | 实时检测 `checkToolInstalled()` | 否 | 自动恢复为未完成 |
| Step 2: 添加供应商 | 实时检测 providers 数量 > 0 | 否 | 自动恢复为未完成 |
| Step 3: 测试切换 | 持久化切换事件标记 | 是 | 保持已完成 |

- **步骤1(安装CLI)**:
  - 每次渲染实时调用 `checkToolInstalled()`，**不调用** `onboardingStorage.addCompletedStep()`
  - 监听 `useToolVersionQuery`，有 version 返回 → 显示完成；version 为空 → 显示未完成
  - 安装成功后自动刷新工具版本查询
  - **若用户后续删除了 npm 包，步骤1 自动恢复为未完成，重新显示「开始安装」按钮**
- **步骤2(添加供应商)**: 实时监听 providers 数量，从 0 变为 >0 时显示完成；降回 0 时恢复未完成
- **步骤3(测试切换)**: 监听 switch 事件，首次切换时调用 `onboardingStorage.addCompletedStep('test-switch')` 持久化标记完成

### 进度条动画

- 使用framer-motion实现平滑过渡
- 每完成一步,进度条增长动画(duration: 0.5s, ease: "easeOut")

### 关闭逻辑

- 点击关闭按钮 → 设置dismissed=true → 整个卡片淡出消失
- dismissed状态持久化,下次打开不再显示
- 如果用户完成所有步骤,自动淡出(延迟2秒,显示"完成🎉"后消失)

### 响应式处理

- 移动端(<640px): 步骤卡片垂直堆叠,隐藏详细描述
- 平板(640-1024px): 紧凑布局,按钮图标化
- 桌面(>1024px): 完整展示

## 设计权衡

### 为什么选择Checklist而非Modal?

- **非阻塞**: 用户可以随时跳过,不强制完成
- **进度可见**: 即使关闭后重新打开,进度仍然保留(如果不dismiss)
- **空间利用**: 在空状态下占用的空间本来就是空的

### 为什么不用Tooltip引导?

- **学习曲线**: 新用户对界面不熟悉,Tooltip需要到处找元素
- **时间成本**: Checklist可以一次性看到所有步骤,Tooltip需要一步步点击
- **适用场景**: Tooltip适合功能发现,Checklist适合首次激活

### 为什么3步而非5步?

- **认知负荷**: 3步是黄金数字,不会让用户觉得复杂
- **核心价值**: 这3步正好覆盖"安装→配置→使用"的最小激活路径
- **完成率**: 步骤越少,完成率越高

## 后续优化方向

1. **A/B测试**: 对比有无引导的激活率差异
2. **热图分析**: 追踪用户在哪一步卡住最多
3. **个性化引导**: 根据用户来源(首次安装 vs 从其他应用迁移)显示不同步骤
4. **视频教程**: 在checklist中嵌入1分钟快速上手视频
5. **智能提示**: 如果用户在某步停留超过1分钟,弹出帮助提示
